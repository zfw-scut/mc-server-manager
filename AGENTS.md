# Codex 项目规范

本文件只用于约束在此仓库工作的 Codex。面向成员的说明位于 `docs/user/`，不要把本文件当作用户教程。

## 开始工作

1. 先阅读 `.codex/rules/remote-operations.md`。
2. 涉及正式服务器时，再阅读 `.codex/context/production.md`。
3. 根据任务阅读 `docs/user/` 和 `docs/history/` 中相关记录，避免重复部署或覆盖既有配置。
4. 连接凭据只从 `.local/servers.yaml` 读取，绝不把其中内容写入响应、日志、提交或命令输出摘要。

## 文档边界

- `docs/user/`：成员可以直接阅读的使用说明和交互方法。
- `docs/history/`：脱敏后的部署、诊断和变更历史。
- `.codex/rules/`：Codex 必须遵守的强制操作规则。
- `.codex/context/`：供 Codex 快速定位服务、目录和端口的技术上下文。
- `.local/`：本机凭据、诊断原件和临时状态，必须保持 Git 忽略。

不要在用户文档中混入内部执行约束，也不要在 Codex 规范中保存密码。

## 工作原则

- 默认把正式 Minecraft 服务视为不可中断服务。
- 状态查询和日志读取可以自主执行；产生远程写入、重启或玩家影响的操作必须符合远程操作规则。
- 面板、脚本、systemd 单元和配置模板应先在仓库中修改，再部署到服务器。避免形成只存在于服务器上的代码。
- 保留用户已有文件和远程配置；不要擅自清理、覆盖或回滚无关内容。
- 修改规模应与任务相称。这个项目服务于 2 至 3 人，不引入不必要的平台、审批层级或基础设施。
- 完成任务后验证目标功能、服务状态和监听端口，并更新脱敏后的历史记录。

## Git 与制品

- 不提交 `.local/`、密码、令牌、私钥、世界存档、备份、整合包 ZIP、Spark 原始报告或运行日志原件。
- 大型二进制制品只记录版本、来源、校验值和外部存放位置。
- 提交信息应说明具体结果，例如 `feat(panel): 显示在线玩家` 或 `fix(backup): 校验上传结果`。
- 紧急远程修复若无法先修改仓库，修复后必须尽快把实际配置同步回仓库并记录原因。
